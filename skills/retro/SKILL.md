# Skill: retro — 週次振り返り・レトロスペクティブ

週単位の進捗・失敗・学びを構造的に振り返り、翌週の改善アクションを1〜3個に絞る。「反省会」ではなく「次をどう変えるか」に集中する。

## トリガー
- 「週次振り返り」「レトロ」「今週の振り返り」
- 「先週どうだった？」「改善点まとめて」
- 「KPT」「Keep/Problem/Try」
- `/retro`

## 実行ステップ

1. **データ収集**（自動取得を試みる）
   - `git log --since="7 days ago" --oneline` でコミット一覧
   - `.queue/done/` の完了タスク数
   - KPI（SNSフォロワー増減、記事公開数、AXIS進捗）

2. **Keep（続けること）**
   - 今週うまくいったこと（最大3点）
   - 数字で示せるなら数字で

3. **Problem（問題だったこと）**
   - 想定より遅れた or うまくいかなかったこと（最大3点）
   - 感情論ではなく「何が起きたか」の事実で

4. **Try（来週試すこと）**
   - 来週変えること（最大3点、具体的なアクションで）
   - 曖昧な「頑張る」は不可。「毎朝9時に/status実行」のように具体的に

5. **翌週の優先度確認**
   - docs/shared-memory/tasks.md を参照して重点タスクを再確認
   - タイムライン（JTB、アジア競技大会）との整合性

## 出力フォーマット

```

**今週のサマリー**
- コミット: X件 | タスク完了: Y件 | SNS: +Z フォロワー

**Keep（続ける）**

**Problem（問題）**

```

## AXISプロジェクト固有の確認

- JTBマイルストーン進捗（LOI/MOU目標5〜6月）
- アジア競技大会まで残り日数
- 在職中/退職後タイムラインとの整合
- セキュリティ・法務リスクの新規発生確認
- キューの滞留タスク（7日超のexpired）整理

## 自動実行（night_worker連携）

毎週月曜3:00のnight_workerが自動でretroを実行し、結果をメールで送信。
手動で呼び出すと即時実行（週中でも可）。

## Gotchas（よくある失敗パターン）

- **Problemで自己批判が多くなる**: 事実を書く。「怠けた」ではなく「X件のタスクが完了しなかった」
- **Tryが抽象的**: 「改善する」ではなく「火曜日に30分、Xをやる」の粒度で
- **Keep/Problemが多すぎる**: 各3点が上限。多くても覚えられない
- **毎週同じTryを書く**: 前回のTryが実行されたかを先に確認。できなかったなら理由を書く
- **JTBタイムラインを見忘れる**: 毎週必ずタイムライン確認を入れる（残り日数を可視化）
