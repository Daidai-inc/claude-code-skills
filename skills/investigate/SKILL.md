# Skill: investigate — 体系的なデバッグ・原因調査

症状から根本原因を体系的に特定する。「なぜ動かないか」ではなく「何が起きているか」から始めて、仮説→検証のループで確実に原因を絞り込む。

## トリガー
- 「なぜ動かない？」「原因を調べて」「バグの原因が分からない」
- 「エラーの意味を調べて」「デバッグして」
- 「再現しない」「本番だけ起きる」「断続的に失敗する」
- `/investigate`

## 実行ステップ

1. **症状の正確な記述**
   - 「何が」「どこで」「いつから」「どのくらいの頻度で」起きているか
   - 期待する動作 vs 実際の動作を1行で
   - エラーメッセージ・スタックトレースの有無

2. **システム境界の特定**
   - 問題が起きているレイヤー（UI/API/DB/外部サービス/インフラ）
   - 最後に動いていた状態からの変更点（デプロイ・設定変更・データ変化）

3. **仮説リスト（最大3つ、確率順）**

   | 確率 | 仮説 | 根拠 |
   |------|------|------|
   | 高 | ... | ... |
   | 中 | ... | ... |
   | 低 | ... | ... |

4. **検証計画**
   - 最高確率の仮説から検証する最小ステップ
   - ログ・DB・APIレスポンスで確認すべき具体的な値
   - 再現手順（再現可能なら半分解決）

5. **根本原因の確定 → 修正方針**
   ```
   根本原因: ...
   修正: ...（コード変更 / 設定変更 / データ修正）
   再発防止: ...（テスト追加 / アラート追加 / ドキュメント更新）
   ```

## AXISプロジェクト固有の調査ポイント

- **Azure App Service**: インスタンス再起動・スロットスワップのタイミングで起きる問題
- **Prisma接続プール**: 接続枯渇（特に Azure の接続数制限）
- **next-auth v5**: セッションCookieの有効期限・JWTデコードエラー
- **Vercel/Azure環境差**: 環境変数の未設定・ビルド時 vs ランタイムの違い
- **CORS/CSP**: 外部API連携（JTB API, AWS S3/SES）でのブラウザエラー

## よくあるパターン（先に確認）

```
1. 環境変数が本番にセットされていない
2. Prismaスキーマとマイグレーションのズレ
3. Azure App Service の warm-up タイムアウト
4. next-auth NEXTAUTH_URL の設定ミス
5. S3/SES IAMポリシーの権限不足
```

## 出力フォーマット

```
## 調査: [症状の一言要約]

**症状**: [期待] vs [実際]
**レイヤー**: UI / API / DB / インフラ

**仮説**
1. [高] ... → 確認: `コマンドや確認箇所`
2. [中] ... → 確認: `コマンドや確認箇所`

**今すぐ確認するコマンド**:
\`\`\`bash
# 最優先の検証コマンドを具体的に
\`\`\`

**根本原因**: （判明後に記入）
**修正**: ...
**再発防止**: ...
```

## Gotchas（よくある失敗パターン）

- **症状を原因と混同する**: 「Prismaエラーが出た」は症状。原因は接続プール枯渇かもしれない
- **仮説を検証せずに修正する**: 確証なしの修正は新しいバグを生む。最小検証を先にやる
- **ログを見ない**: エラーメッセージを読まずに推測するのは非効率。まずログ確認
- **環境差を無視する**: 「ローカルで動く」は情報。本番だけ起きるなら環境差を先に疑う
- **1つ直したら完了と思う**: 根本原因が複数の症状を持つことがある。関連する問題を一緒に確認
