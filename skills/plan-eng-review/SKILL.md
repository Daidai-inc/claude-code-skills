# Skill: plan-eng-review — エンジニアリング視点での設計レビュー

実装計画・技術選定・アーキテクチャをEngineering Manager視点でレビューし、設計の穴・テスト戦略・リスクを特定する。/plan-ceo-review（Why/What）の後段として使い、How（どう作るか）を固める。

## トリガー
- 「エンジニアレビューして」「設計を見て」「アーキテクチャ確認」
- 「実装前にレビュー」「技術的に問題ない？」
- 「テスト戦略を考えて」
- `/plan-eng-review`

## 想定インプット例
- AXIS新機能の実装計画
- Azure/Next.js/Prismaのアーキテクチャ変更
- データモデル設計
- API設計・外部連携仕様
- パフォーマンス・スケーリング要件

## 実行ステップ

1. **対象の確認**（1〜2文で要約）
   - 何を作るか・変更するかを1文で
   - 影響するシステム境界（DB/API/UI/外部サービス）を確認

2. **アーキテクチャ・データフロー確認**
   - 主要コンポーネントと依存関係
   - データの流れ（入口→処理→出口）
   - 状態管理・キャッシュ・非同期処理のリスク

3. **設計の問題点（最大5点）**

   | 重要度 | 問題 | 修正案 |
   |--------|------|--------|
   | 高/中/低 | ... | ... |

4. **テスト戦略**
   - 単体テストが必要なロジック
   - 統合テストが必要な境界
   - E2Eテストのシナリオ（最重要ユーザーフローのみ）

5. **判定**
   ```
   判定: [実装GO / 設計修正してGO / 再設計必要]
   ブロッカー: （あれば。これが解決しないとGO出せない）
   推奨アクション: （今すぐやること）
   ```

## AXISプロジェクト固有の確認ポイント

- Next.js App Router/RSC: Server/Client境界は適切か
- Prisma: N+1クエリ・トランザクション境界
- Azure App Service: スケールアウト時の状態共有問題
- next-auth v5: セッション管理・ミドルウェアとの競合
- AINAGOC情報管理規程: 選手・宿泊データの暗号化・アクセス制御

## 出力フォーマット

全体で20〜25行以内。

```
## エンジニアレビュー: [対象の一言要約]

**アーキテクチャ**
- データフロー: ...
- 境界: ...

**設計の問題点**
- [高] ... → 修正案: ...
- [中] ... → 修正案: ...

**テスト戦略**
- 単体: ...
- 統合: ...

**判定**: 実装GO / 設計修正してGO / 再設計必要
**ブロッカー**: （なし or 具体的な問題）
**今すぐやること**: ...
```

## Gotchas（よくある失敗パターン）

- **完璧な設計を求めすぎる**: 一人開発では"十分に良い"設計で進む。完璧主義は期限を殺す
- **テスト戦略なしでGOを出す**: 最低限E2E1本は指定する（後で書くは幻想）
- **Azure固有の問題を見落とす**: Vercelと違いスケールアウト時にインメモリ状態は消える
- **外部連携（JTB API等）を楽観視する**: タイムアウト・レート制限・障害時のフォールバックを確認
- **Prismaマイグレーションのリスクを過小評価**: 本番DB変更は必ずロールバック手順を先に確認
